## 개요

SOOL 프로젝트의 인증(Auth) 시스템은  
**MVP 단계에서는 최소화**,  
**확장 단계에서는 자연스럽게 추가**하는 전략으로 설계되었다.

현재 단계에서는 인증 없이도
전통주 탐색·분석이 가능하도록 하여
사용자 진입 장벽을 낮추는 것을 목표로 한다.

---

## MVP 단계 정책 (현재)

### 인증 상태
- ❌ 로그인 없음
- ❌ 회원가입 없음
- ⭕ 모든 데이터는 공개 조회 가능

### 이유
- 데이터 탐색 중심 서비스
- 추천/리뷰 기능 이전 단계
- 초기 사용성 확보가 최우선

---

## 사용자 흐름 (MVP)

비회원 사용자
↓
전통주 탐색
↓
상세 정보 조회
↓
데이터 기반 이해


---

## 인증 도입 시점 (Phase 2)

다음 기능이 필요해지는 시점에 인증을 도입한다:

- 리뷰 작성
- 즐겨찾기
- 개인화 추천
- 데이터 기여

---

## 인증 구조 설계 (예정)

### 사용자 테이블

| 컬럼 | 타입 | 설명 |
|---|---|---|
| id | INTEGER | PK |
| email | TEXT | 로그인 ID |
| password_hash | TEXT | 암호화 비밀번호 |
| created_at | DATETIME | 가입 시각 |

---

### 인증 방식

- JWT 기반 인증
- Access Token / Refresh Token 구조
- FastAPI dependency로 인증 처리

---

## API 확장 예시

POST /api/auth/register
POST /api/auth/login
GET /api/users/me


---

## 보안 고려사항

- 비밀번호 해시 처리 (bcrypt)
- 토큰 만료 시간 관리
- 인증 API rate limiting
- 공개 API와 인증 API 분리

---

## 프론트엔드 연동 계획

- 로그인/회원가입 전용 페이지
- 인증 상태에 따른 UI 분기
- 토큰은 HttpOnly Cookie 또는 Secure Storage 사용

---

## 설계 철학 요약

- **지금 필요한 것만 구현**
- 인증은 기능이 아니라 **시점의 문제**
- 데이터 중심 서비스 흐름을 해치지 않는 방향

---

## 요약

SOOL의 인증 설계는  
“미리 만들어 두지 않지만,  
언제든 자연스럽게 붙일 수 있도록”  
구조를 열어두는 전략을 따른다.
